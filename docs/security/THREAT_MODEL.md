# Threat Model

This source-review threat model is not a replacement for an operational risk assessment. Severity reflects residual risk after controls observed in this repository.

| Threat | Asset | Entry point | Existing control | Remaining risk | Severity | Recommended action | Owner | Status |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| Credential guessing or user enumeration | User accounts | `POST /auth/login` | bcrypt, generic invalid-credential message, timing-safe dummy compare, failed-login lockout | No IP/device rate limit; distributed attacks can continue | High | Add reverse-proxy or application rate limiting and alerting | Platform owner | Deferred |
| Session theft | Authentication tokens | Cookies, browser/device | HttpOnly, SameSite=Lax, production requires Secure, refresh rotation and hashed storage | XSS or compromised device can still act as the signed-in user; access token lasts until expiry | High | Enforce HTTPS/HSTS at the edge; add token-revocation strategy if immediate logout is required | Platform owner | Partially mitigated |
| Cross-site request or UI embedding | Authenticated actions | Browser requests/frames | CORS configured to one origin, credentials enabled, SameSite=Lax, anti-framing headers | SameSite is not a full CSRF defence; future cross-site requirements need explicit CSRF design | Medium | Keep one trusted origin; add CSRF tokens before using `SameSite=None` or cross-site clients | API owner | Deferred |
| Unauthorized record access/edit | FG records and approvals | Record and workflow endpoints | Global JWT, role/permission guards, owner checks, record lock state, audit logs | New endpoints may omit decorators; auditor read-only behavior is policy-derived rather than separately exercised end-to-end | High | Add controller authorization integration tests and maintain an endpoint permission matrix | API owner | Partially mitigated |
| Privilege escalation or unsafe approval | Loading decisions | `POST /inspection-records/:id/loading-decision` | Role and permission gates; critical failure cannot be overridden to approval; approval/audit rows | Check/Verify workflow is not implemented and must not be assumed | Medium | Implement Check/Verify only from approved requirements, with dedicated authorization tests | Product and QA owner | Deferred |
| Oversized or malicious evidence input | Database availability, reviewers | Draft/submit response payloads | Four attachments per response; JPEG/PNG/WebP base64-only validation; 5 MiB decoded cap; default request body limit | Data URLs are stored in the DB; no malware scanning or content decoding | High | Replace with object storage plus quarantine and malware scanning before broad production use | Platform and security owner | Partially mitigated |
| Script injection | Browser session and records | Rendered text, evidence URLs | React escaping; no dangerous HTML found; CSP, `nosniff`, restricted evidence MIME | CSP permits inline scripts/styles for Next compatibility | Medium | Move to nonce/hash CSP when compatible; keep user content out of HTML sinks | Web owner | Partially mitigated |
| Dependency compromise | Build/runtime environment | Package installation and transitive dependencies | Lockfile, audit executed | Audit reports vulnerable tar and postcss versions | High | Upgrade parent dependencies in a tested maintenance change; do not force incompatible transitive major upgrades | Dependency owner | Deferred |
| Secret exposure | Database/JWT credentials | Repository, bundles, CI/CD | `.env` ignored, production JWT env required, tracked-file scan | `.env.example` includes intentionally insecure development values; deployment secret management is not represented | Medium | Use a managed secret store, rotate dev defaults, and scan CI artifacts | Platform owner | Deferred |
| Lost or shared device exposure | Draft record data | Browser localStorage | Drafts cleared after successful submit; auth cookies are HttpOnly | LocalStorage is readable by anyone using an unlocked device and persists across logout/browser restarts | Medium | Use managed devices, screen locks, user training, and provide a clear-local-drafts action at sign-out | Operations owner | Deferred |
| Error-information disclosure | Internal implementation details | API failures and logs | Framework HTTP exception handling, safe custom auth errors | No explicit production exception filter or tested error response policy | Medium | Add a production exception filter with generic responses and correlation IDs | API owner | Deferred |
